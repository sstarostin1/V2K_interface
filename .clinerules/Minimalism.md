# Принцип минималистичной архитектуры

## Основное правило
Все архитектурные решения должны следовать принципу минимальной достаточности. Создавай ровно столько файлов и модулей, сколько необходимо для решения задачи, но не больше.

## Критерии принятия решений

### 1. Обоснование создания нового файла
Новый файл создается ТОЛЬКО когда:
- Функциональность явно отделена и может использоваться независимо
- Логическое разделение улучшает читаемость и поддерживаемость
- Разные команды работают над разными частями системы
- Требуется изоляция сложной или специфической логики

### 2. Приоритеты в организации кода
- **Монолит над микросервисами** для небольших проектов
- **Функции в одном файле** над разбиением на множество мелких файлов
- **Параметризация** над созданием отдельных реализаций
- **Композиция** над наследованием, где это уменьшает сложность

### 3. Запрещенные практики
- Создание файлов-"заглушек" без четкой текущей необходимости
- Разделение кода только по "возможному будущему использованию"
- Добавление абстракций там, где прямое решение проще и понятнее
- Создание конфигурационных файлов для тривиальных настроек

### 4. Процедура проверки перед созданием файла
Перед созданием каждого нового файла задай себе:
- Можно ли эту функциональность разместить в существующем файле?
- Упростит ли это разделение понимание кода?
- Будет ли этот файл использоваться независимо?
- Соответствует ли это разделение фактической структуре данных?

## Метрики контроля
- Среднее количество файлов на 1000 строк кода должно быть минимальным
- Глубина вложенности директорий не должна превышать 3-4 уровней
- Коэффициент повторного использования кода должен быть максимальным
- Каждый файл должен содержать логически связанную функциональность

## Исключения
Разрешается создание дополнительных файлов только для:
- Генерации кода или автоматизации сборки
- Тестов, которые требуют изоляции
- Документации и конфигурации deployment
- Внешних интеграций с четко отделенным контрактом

Помни: простота архитектуры - признак зрелости решения, а не недостаток проработки.
