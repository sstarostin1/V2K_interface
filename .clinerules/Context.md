# Процедура первичного анализа проекта

## Активация
Срабатывает только при первом обращении к модели в новом сеансе, когда отсутствует контекст предыдущих обсуждений.

## Протокол анализа

### 1. Быстрый обзор структуры
- Проверь наличие стандартных директорий проекта:
  - `/DOCS` - документация (если есть, используй для быстрого понимания)
  - `Neuro_Deleted_Archive` - архив удаленных компонентов (история изменений)
  - `/.clinerules` - локальные правила проекта (уже загружены в контекст)
  - `/workflows` - рабочие процессы (если применимо)

### 2. Стратегия исследования
- **Используй документацию**: Начни с `/DOCS` для общего понимания архитектуры
- **Проверь архив удалений**: Изучи `Neuro_Deleted_Archive/README.md` для понимания истории изменений
- **Фокус на релевантности**: Читай только те файлы исходного кода, которые непосредственно связаны с текущим запросом
- **Игнорируй .clinerules**: Правила уже загружены, не трать токены на их повторное чтение

### 3. Экономное чтение кода
- При большом проекте читай только:
  - Основные конфигурационные файлы
  - Файлы, непосредственно упомянутые в запросе
  - Ключевые модули, связанные с текущей задачей
- Используй документацию для понимания общей структуры вместо чтения всего кода

### 4. Контекстуализация
- Учитывай, что запрос может быть продолжением предыдущего обсуждения
- Используй артефакты проекта (документацию, архив) для восстановления контекста
- Выявляй связи текущей задачи с существующей функциональностью через документацию

## Выходные данные
После анализа предоставь краткий отчет:
- Общее понимание проекта на основе документации
- Ключевые моменты, обнаруженные в релевантных файлах
- Вопросы для уточнения, если что-то осталось неясным

## Особые случаи

### Обращение к .clinerules
- Не читай файлы из `.clinerules` при первичном анализе
- Если оператор явно ссылается на конкретное правило или роль - можешь обратиться к соответствующему файлу
- В остальных случаях предполагай, что правила уже учтены в твоей работе

### Использование архива удалений
- Проверяй архив перед удалением или изменением кода
- Используй историю изменений для понимания эволюции проекта
- Учитывай ранее удаленные решения при предложении новых

## Важные замечания
- Экономь токены: используй документацию вместо чтения всего кода
- Фокусируйся на областях, непосредственно связанных с запросом
- Помни о возможной потере контекста из-за ограничения длины окна
- Запрашивай уточнения при обнаружении противоречий между кодом и документацией

Этот анализ - основа для последующей продуктивной работы над проектом.
